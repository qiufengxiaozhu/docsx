# 部署优化 - 资源压缩

离线版体积不小：`x2t.wasm` 40M、各编辑器 `sdk-all.js` 11~15M、字体库 327M。不做任何处理的话，首次打开编辑器要下载上百 MB，慢到不可用。

处理思路只有两条，都不需要改代码：

- **预压缩**：部署前生成 `.br` / `.gz` 文件，传输体积降到原来的 1/5 左右；
- **强缓存**：资源路径自带版本号，可以一年 `immutable` 缓存，回头客零下载。

## 1. 压缩效果实测

brotli / gzip 实测数据（brotli -q 11，gzip -9）：

| 文件 | 原始 | brotli | gzip |
|------|-----:|-------:|-----:|
| `common/wasm/x2t/x2t.wasm` | 40M | **6.6M（-84%）** | 9.4M（-77%） |
| `pdf/src/engine/drawingfile.wasm` | 9.6M | **3.1M** | 4.0M |
| `common/libfont/engine/fonts.wasm` | 3.4M | **1.0M** | 1.4M |
| `word/sdk-all.js` | 14M | **2.1M** | 2.8M |
| `cell/sdk-all.js` | 15M | **2.4M** | 3.1M |
| `vendor/fonts/*`（327M，按需加载） | 单个最大 16M | **-51%**（q9） | -45% |

wasm 和未压缩的 JS 都是高度可压缩格式，预压缩是所有优化里收益最大的一步。

关于字体：`vendor/fonts/` 下是按索引重命名的无扩展名字体文件（`000`、`001`…），**按需加载**——文档用到哪个字体才请求哪个，不占首屏。压缩后配合强缓存，二次打开零请求。

## 2. 生成预压缩文件

项目根目录提供了 `precompress.sh`，为大于 100K 的 `.wasm` / `.js` 及 `vendor/fonts/` 下的字体生成同名 `.br` 和 `.gz`（放在原文件旁边）：

```bash
# 依赖 brotli：
#   macOS:         brew install brotli
#   Debian/Ubuntu: apt install brotli
#   CentOS:        yum install brotli
./precompress.sh
```

说明：

- **增量执行**：源文件没变则跳过，重复运行无代价；
- **压缩级别**：js/wasm 用 q11（追求极限压缩率），字体量大改用 q9（压缩率几乎不变，速度快一个量级）；
- **在哪跑都行**：本机跑完随包部署，或上传后在服务器跑（上行带宽紧张时推荐后者，少传 300M+ 压缩产物）。

> **为什么预压缩，而不是 nginx 运行时压缩（`gzip on`）？**
> 40M 的文件每次请求现压一遍 CPU 开销太大，且运行时只能用低压缩级别。预压缩后 nginx 把磁盘上的 `.br`/`.gz` 直接发出去：零 CPU 开销，压缩率还最高。

## 3. Nginx 配置

```nginx
server {
    listen 80;
    server_name example.com;

    root /path/to/OnlyofficePersonal;
    index office.html;

    # 请求 x2t.wasm 时，按客户端 Accept-Encoding 直接返回 x2t.wasm.br / x2t.wasm.gz
    brotli_static on;   # 需要 ngx_brotli 模块，见下方说明
    gzip_static  on;    # nginx 原生支持，作为兜底

    # 路径带版本号（9.3.0.135-xxx），升级版本路径必变，可以放心永久缓存
    location ~* \.(wasm|js|css|woff2?|png)$ {
        add_header Cache-Control "public, max-age=31536000, immutable";
    }

    # 字体文件按索引命名、无扩展名，缓存头单独给一条
    # （brotli_static 按「URI + .br」查找文件，对无扩展名文件同样生效，无需额外配置）
    location ~ ^/9\..*/vendor/fonts/ {
        add_header Cache-Control "public, max-age=31536000, immutable";
    }
}
```

两个注意点：

- **wasm 的 MIME 必须是 `application/wasm`**，否则浏览器无法流式编译（`instantiateStreaming`）。nginx 1.21+ 的 `mime.types` 已内置，老版本需手动添加；
- **`brotli_static` 依赖 [ngx_brotli](https://github.com/google/ngx_brotli) 模块**，官方 nginx 不自带。检查：`nginx -V 2>&1 | grep -o brotli`。

没有 ngx_brotli 的两种解决方式：

1. **只开 `gzip_static`**：nginx 原生支持，wasm 也能压掉 77%，够用；
2. **用带 brotli 的 Docker 镜像**，不动宿主机环境：

```bash
docker run -d --name onlyoffice-static --restart unless-stopped \
  -p 8080:80 \
  -v /path/to/OnlyofficePersonal:/usr/share/nginx/html:ro \
  -v /path/to/default.conf:/etc/nginx/conf.d/default.conf:ro \
  macbre/nginx-brotli
```

## 4. 验证

```bash
# 应返回 Content-Encoding: br（或 gzip）
curl -sI -H 'Accept-Encoding: br' \
  http://example.com/9.3.0.135-24e34b4f401d86dfb31def637358e2fa/vendor/sdkjs/common/wasm/x2t/x2t.wasm \
  | grep -iE 'content-(encoding|length|type)'
```

预期输出：

```
content-type: application/wasm
content-encoding: br
content-length: 6898179        # ≈6.6M 而不是 40M
```

浏览器验证：DevTools → Network 看 `x2t.wasm` 的 Size 列，传输体积 ~6.6M、资源体积 40M 即生效；二次刷新显示 `(disk cache)` 说明缓存也生效了。

## 5. 缓存说明

压缩解决**首次**加载，缓存解决**后续**加载：

- 资源都在带版本号的目录下（`9.3.0.135-24e34b4f...`），内容变化必然伴随路径变化，因此 `immutable` + 一年缓存是安全的；
- 生效后回头客加载编辑器不产生任何资源请求（磁盘缓存直出），打开过的文档字体也同理。

## 6. 其他静态服务器

| 服务器 | 做法 |
|--------|------|
| Caddy | `file_server { precompressed br gzip }`，原生支持 |
| Go 后端 | 检查 `Accept-Encoding`，存在 `.br` 文件则返回它并设 `Content-Encoding: br` |
| GitHub Pages / 对象存储 | 无法自定义预压缩，依赖 CDN 自动压缩（通常只有 gzip，大文件可能被跳过） |

## 7. 常见问题

**能不能用 wasm-opt 给 x2t.wasm 瘦身？**
不建议。`x2t.wasm` 是官方编译产物，`wasm-opt` 收益（10~20%）远小于传输压缩（84%），还有改坏的风险。

**能不能删掉不用的字体，让 327M 变小？**
不建议。字体按索引注册在 `AllFonts.js` 里，裁剪需要重建索引，容易把字体选择功能搞坏。字体本身按需加载，不影响首屏，压缩 + 缓存已经够了。

**`.br` / `.gz` 文件要不要提交进仓库？**
二选一：提交进去则部署机不需要装 brotli（代价是仓库大 ~300M）；不提交则在 `.gitignore` 加 `*.br` / `*.gz`，部署时跑一次 `precompress.sh`。

**gzip 和 brotli 的文件都要生成吗？**
建议都生成。现代浏览器都支持 brotli，但 `Accept-Encoding` 里没有 `br` 的老客户端（或某些代理后面的请求）会回退到 gzip。
